00主线：一个模型如何被"逼"成能干活

这些板块不是并列学科，而是同一条能力演化主线上的不同重心，外加贯穿全程的效率与评估底座。

大模型算法的所有努力，本质在回答一个问题：怎么把一个只会"预测下一个 token"的概率模型，逼成"能可靠完成复杂任务"的系统。能力沿"生成 → 释放 → 扩展"递进；而推理效率、评估这两件事像地基，托着上面所有方向。

01预训练（基模）：底座能力从哪来

技术本质：在自回归（预测下一个 token）这个极简目标下，靠规模把通用能力压进参数。所有难点都在"如何把规模与数据工程化"，而非那个 loss 公式本身。

1.1 架构：Transformer 之上的工程化分化

主干仍是 Decoder-only Transformer，但近年的关键创新全在"如何在固定算力下塞进更多有效能力、并能推得起"。逐层看：

Dense vs MoE
MoE（混合专家）把 FFN 层换成多个"专家"，每个 token 经路由只激活 Top-K 个——总参数可达万亿级，实际计算只动一小部分。难点：① 路由负载均衡（加 auxiliary loss 或用 loss-free 的偏置调节，防止少数专家被挤爆）；② 训练不稳；③ 推理时专家分布在多卡，通信开销大。是当前冲规模的主路线。
注意力变体
MHA→GQA→MQA 是"查询头共享多少组 KV"的谱系，越共享越省 KV cache、推理越快，但表达力略降，GQA 是当前折中主流。更新的 MLA（多头潜在注意力）把 KV 压成低秩潜向量，进一步省显存。
归一化与残差
普遍用 Pre-Norm（norm 放在子层前）保证深层训练稳定；RMSNorm 取代 LayerNorm（去掉均值项，更快）。这些细节直接关系到能不能把模型训到几十上百层不发散。
激活函数
FFN 里 SwiGLU（门控+SiLU）几乎成了标配，比 ReLU/GeLU 收敛更好。看似小改动，但在大规模上累积出明显增益。
位置编码与长程
RoPE（旋转位置编码）是主流；扩长上下文靠对 RoPE 频率做插值 / NTK 缩放 / YaRN。真正难的不是"塞得进 128K"，而是塞进去后"中间段信息被忽略"（lost-in-the-middle）和注意力被稀释。

1.2 数据：预训练真正的胜负手

架构趋同后，拉开模型差距的主要是数据。这条流水线比想象的重得多：

  • 配比 mixture：中/英/代码/数学/学科语料的比例直接决定能力画像。一个反直觉但已被反复验证的结论——大量代码与数学语料能显著提升通用推理，所以做中文模型也要喂海量代码。
  • 质量与去重：大规模语义级去重（MinHash/SimHash + 近似聚类，而非字面匹配）、质量分类器打分、毒性与隐私过滤。"垃圾进垃圾出"在这里是字面意义。
  • 课程与退火：训练并非均匀采样。常用"课程学习"由易到难，末期用 annealing（学习率衰减阶段换上最高质量数据收尾），对最终质量影响巨大。
  • 合成数据：高质量真实语料逼近"数据墙"后，用强模型生成训练数据成为最热也最有争议的方向——风险是"模型近亲繁殖"导致分布坍缩（model collapse）。
  • 数据污染：训练集若混入评测集，会让 benchmark 虚高。去污染（decontamination）是发布可信模型的必备步骤，也与第 06 节评估强相关。

1.3 Tokenizer：常被忽视但影响深远

模型不认字，先用 BPE/Unigram 把文本切成 subword token。词表设计直接影响：中文与多语言的压缩率、数字怎么切（按位切对算术更友好）、罕见词与代码符号的处理。词表一旦定下、预训练开跑就难改，是"地基中的地基"。

1.5 规模规律与"能力涌现"

Scaling Law 决定"给定算力，模型多大、数据多少最优"。Kaplan 早期偏大模型，Chinchilla 修正为"数据其实喂少了，参数与数据应同比增长"；如今因推理成本敏感，转向"小模型 + 过度训练"（喂远超 Chinchilla 最优的数据量，换更便宜的推理）。涌现能力（某些能力到一定规模才突然出现）是否真实、还是评测指标的错觉，仍是开放争论。

前沿张力

① 真实高质量语料逼近"数据墙" → 合成数据 / 数据效率 / 多 epoch 复用成焦点；② 推理成本主导决策 → 架构走 MoE + 小激活、训练走过度训练；③ 长上下文从"能读"卷向"读得准"，并与下文的 KV cache 效率深度耦合。

金融落地预训练在金融的切口主要是继续预训练（continual pretraining）：在通用底座上灌入金融语料（研报、财报、招股书、合规条款、行情与交易数据）做领域增强，而非从零自研。数据配比要补强数字/表格/时序语料以提升数值推理；Tokenizer 对金额、比率、长数字的切分方式直接影响算术准确性；长上下文能力对应"读完整份财报/合同再回答"的刚需。

02后训练 / 对齐：RL 为什么成了主战场

技术本质：预训练把能力"存"进去了，但模型不知道何时、以何种方式取出来。后训练做两件事——激发（按指令/格式/推理链输出）与对齐（符合偏好与安全约束）。这一阶段是当前创新最密集处。

2.1 SFT：模仿学习，质量压倒数量

用高质量"指令—回答"对做监督微调。要点比看上去深：

  • 数据质量：已成共识——几千到几万条精挑数据，胜过百万条噪声。多样性和难度分布比总量重要。
  • 拒绝采样 rejection sampling：让模型自己生成多个答案，用奖励/规则筛出最优再回灌训练。是 SFT 与 RL 之间的桥梁，也是"自我提升"的雏形。
  • 样本打包与损失掩码：多条短样本 packing 进一个序列提效率；只对"回答"部分算 loss（mask 掉指令），避免模型学着复述提问。

2.2 奖励建模 RM：整条 RLHF 的天花板

训一个模型预测"人类更偏好哪个回答"，多用 Bradley-Terry 框架从成对偏好里学标量分数。两条关键分化：

类型奖励粒度特点
ORM 结果奖励只看最终答案好坏标注便宜，但对长推理"知其然不知其所以然"
PRM 过程奖励给推理每一步打分对数学/推理显著更优，但步级标注昂贵、难规模化

RM 自身的脆弱性是核心隐患：它只是人类偏好的代理，一旦被过度优化就会偏离真实意图（即下文的 reward hacking）。

2.4 推理模型：RLVR 如何"训"出长思维链

这是近一年最大的范式跃迁。传统 RLHF 对齐的是"人类偏好"（主观），而 RLVR 对齐的是"客观正确"——在数学、代码这类答案可自动验证的任务上，只要答对就给奖励。由于奖励无噪声、可大规模生成，模型在 RL 中自发学会更长的推理链、自我检查、回溯纠错（所谓 test-time scaling：推理时多想，正确率随之上升）。难题随之而来：如何把这套搬到"没有标准答案"的开放任务上，是当前最硬的开放问题。

KL 约束
RL 训练时拉一根"皮筋"，防止模型为讨好奖励而偏离原始分布太远（否则会胡言乱语或丢失通用能力）。
reward hacking
模型钻奖励漏洞拿高分而非真做好——如废话凑长度、迎合 RM 的偏见。RL 最顽固的隐患。
对齐税
alignment tax：对齐/安全约束往往让模型在某些通用能力上略降，如何减小这个代价是持续课题。
test-time scaling
推理时让模型多想几步/多采样几条再选，用"想得更久"换正确率，是推理模型的核心红利。

金融落地后训练是金融大模型落地最吃重的一环。金融是强正确性、强合规领域，恰好契合 可验证奖励（RLVR）+ 过程奖励（PRM）：利息/收益测算、合规判断、引用准确性都能构造客观可验证信号，比单纯对齐人类偏好更可靠。安全对齐要额外叠一层"不得承诺收益、不得提供违规建议"的合规红线对齐；压低幻觉（编造数据/条款）是金融后训练的头号 KPI。

03推理与效率：托起一切的工程地基

技术本质：训出来只是一半，能否便宜、快速、稳定地"跑起来"决定了模型可不可用。这一块独立成一门重活，且与架构、长上下文深度耦合——本版把它从附属提为独立板块。

3.1 自回归推理的根本瓶颈

大模型逐 token 生成，且每步都要回看前文，因此瓶颈不在算力而在显存带宽（要反复搬运 KV）。围绕这个瓶颈，衍生出一整套优化：

KV cache
把已生成 token 的 Key/Value 缓存下来避免重算。它是推理提速的关键，但显存占用随上下文线性增长——长上下文之所以贵，主因就在这。
PagedAttention
像操作系统管内存那样分页管理 KV cache（vLLM 的核心），大幅减少碎片、提升吞吐。配合 continuous batching 动态拼批。
量化
把权重/激活从 16 位压到 8 位甚至 4 位（GPTQ/AWQ 等），省显存、加速，代价是少量精度损失。是落地部署的标配。
投机解码
speculative decoding：用一个小"草稿模型"先快速猜几个 token，大模型一次性并行校验，平均每步省时。无损加速的主流手段。
知识蒸馏
用大模型当老师，把能力"教"给小模型——小模型推理更便宜。当前小模型变强，蒸馏功不可没。
Prefill vs Decode
推理分两阶段：prefill（并行处理输入提示，算力密集）和 decode（逐 token 生成，带宽密集）。两阶段特性不同，催生 P/D 分离部署等优化。

为什么把它单列一节

推理效率不是"训练完之后的小事"——它反向决定架构选型（为什么用 GQA/MLA、为什么用 MoE 的小激活、为什么追求小模型），也决定长上下文、推理模型这些方向能不能真正落地。算法与系统在这里高度交织。

金融落地金融场景对效率的诉求很具体：① C 端高并发（如微信支付级流量）下，量化 + 投机解码 + PagedAttention 决定单位成本扛不扛得住；② 用蒸馏出的小模型承接高频简单咨询、大模型只接复杂决策，是常见的成本分层；③ 读长财报/合同的长上下文场景，KV cache 显存是主要瓶颈，直接影响能不能一次性塞下整份文档。

04Agent 算法：从"会说"到"会做事"

技术本质：让模型在"观察→思考→行动→再观察"的多步循环里可靠完成长链条任务。这里讲的是怎么训练模型本身具备这种能力，而非在外面搭运行框架。

4.2 train-based 的四大核心难题

  • 轨迹数据：需要大量"完整任务执行过程"（含工具调用、中间观察、成败结果）。真实轨迹稀缺，多靠强模型在沙盒里 rollout 生成，质量与多样性难保证。
  • 长程信用分配：任务几十步，最后失败，到底哪一步错的？把稀疏的最终奖励正确分配回中间步骤，是 agent RL 最难的技术点；过程奖励、步级 critic、回溯式归因都是尝试方向。
  • 工具学习：不只是会生成调用格式，而是学会"何时该调、调哪个、参数怎么填、返回报错怎么恢复"。错误处理与重规划能力尤其关键。
  • 环境与评估：要可交互、可重置、可大规模并行的沙盒供反复试错；评估要做到"任务级"（任务有没有真完成），而非只看单轮回答像不像。

前沿张力

把 RLVR 外推到 agent（任务完成即奖励），训练长程、多轮、带真实工具的策略；context 工程（如何在有限窗口里维持长任务记忆）与多智能体协作训练同步起步。本质难点是：真实世界任务的奖励既稀疏又难验证，这让 RLVR 的红利无法直接复制。

金融落地金融 Agent 的特殊性在于动作有真实后果且强监管：查账户、算还款、发起转账、办业务，错一步代价高。因此除了算法侧训规划/用工具能力，落地必须叠"人在回路确认 + 高危动作熔断 + 全程可审计"。可验证奖励在金融 agent 上反而更可得（业务是否办成、金额是否正确可判定），是相对适合用 RL 训练的垂类。

05多模态：把感知接进认知

技术本质：让语言模型处理非文本信息。一个根本分叉必须分开看——理解和生成是两套技术、两条技能栈。

5.1 理解：看懂图/视频/音再用语言回应

主流结构是 视觉编码器 + 连接器 + 语言大模型：图像经编码器（多基于 ViT/CLIP）变成视觉 token，经连接器（projector，可以是简单 MLP，也可以是 Q-Former 这类重采样器）映射进语言模型的表示空间。训练两阶段——先图文对齐预训练（海量图文对让两个空间对上），再多模态指令微调（学会按指令答关于图的问题）。

关键难题

  • 视觉 token 爆炸：高分辨率图、长视频会产生海量视觉 token，挤占上下文且拖慢推理，催生 token 压缩/重采样技术。
  • 细粒度对齐：OCR、小目标、图表数字、密集文档的精确理解，是当前最难啃也最高价值的部分。
  • 模态干扰：接入视觉后纯文本能力退化（"看懂了但变笨了"），需精细配比缓解。
  • 原生多模态：从"后期拼接编码器"走向"预训练阶段就混合多模态一起训"，对齐更深，是演进方向。

5.2 生成：另一套体系

文生图/视频主力是扩散模型：从纯噪声一步步"去噪"还原图像。架构从 U-Net 转向 DiT（Diffusion Transformer），flow matching 让训练与采样更快更稳。也有走自回归路线（把图像离散成 token 像文本一样生成）的流派。与"理解"几乎是两拨人的技能栈。

ViT
视觉版 Transformer，把图片切成 patch 当 token，让模型像读文字一样读图。
CLIP 对比学习
用海量图文对做对比训练，让"图"和"描述它的文字"在表示空间靠近——图文互通的基石。
连接器 / Q-Former
把视觉特征对齐并压缩进语言空间的桥接模块，决定视觉信息进入大脑时的质量与数量。
扩散 / DiT
去噪式生成范式；DiT 用 Transformer 替换 U-Net 骨架，更易随规模扩展，是当前文生图/视频主流。

金融落地金融的高价值区在理解侧而非生成侧：身份证/银行卡/票据/合同的 OCR 与结构化抽取、研报图表与 K 线解读、报表数字提取——核心难点正是上面说的"细粒度对齐"（数字、小字、密集表格不能错）。生成侧（文生图/视频）匹配度低，仅投教内容等少数场景用得上。

06搜推 / RAG：检索体系与大模型的融合

技术本质：从海量候选里高效挑出"最该返回的"并排好。它有自成一体的工业体系，与 LLM 是两条历史线，如今在向量化与 RAG 上深度融合。

6.2 RAG：检索增强生成的工程纵深

RAG 是搜推与大模型结合最成熟的形态：回答前先检索相关资料，再让模型基于资料作答，从而压制幻觉、可溯源、能用专有/最新数据。看似简单，工程链路其实很深：

  • 切块 chunking：文档怎么切——切太碎丢上下文，切太大引入噪声；语义切分、重叠窗口、层级结构都是讲究。
  • 检索：稠密向量检索（懂语义）+ 稀疏检索（BM25，懂关键词）混合，往往优于单一路线。
  • 重排 rerank：用 cross-encoder 对检索结果精排，把最相关的放前面，对最终质量影响极大。
  • 查询改写 / 多跳：把用户口语问题改写成更好检索的形式；复杂问题拆成多步检索（multi-hop）。
  • Agentic RAG：让模型自主决定"要不要检索、检索什么、够不够再来一轮"，是 RAG 与 Agent 的合流方向。
双塔模型
用户与物品各自编码成向量、看谁更配，召回阶段经典结构，支持离线建索引。
ANN 近似检索
在海量向量里快速找"最接近"的（HNSW/IVF 等索引），让搜索从对字面升级到对语义。
Embedding 模型
把文本/物品映射成语义向量，是语义召回与 RAG 的共同底座，其质量直接决定检索上限。
生成式推荐
给每个物品一个语义 ID，把推荐建模成"生成下一个物品 ID"的序列问题，让 LLM 直接生成——新范式，仍在探索。

金融落地两个落点：① 产品推荐——按风险偏好/资产状况匹配理财、贷款、保险，本质是带强合规约束（适当性管理：不能给保守客户推高风险品）的搜推；② RAG 作事实底座——金融问答先检索权威条款/最新行情/产品说明再作答，是压幻觉、可溯源、满足合规留痕的关键形态，往往比"把知识训进模型"更安全可控。

07评估：决定上面一切是否可信

技术本质：没有可靠评估，所有"提升"都是自欺。评估之难，丝毫不亚于训练本身，且随能力变强而更难。
  • 基准污染：评测集泄进训练集让分数虚高，是最普遍的陷阱；需要严格去污染与持续更新的"防泄露"评测集。
  • LLM-as-judge：用强模型当裁判给回答打分，规模化但有偏（偏好长答案、偏好自己的风格、位置偏置），需校准。
  • 能力 vs 安全：既要测能力（推理、知识、代码），也要测安全/对齐（拒答、越狱、偏见、幻觉率），两类评测方法论不同。
  • 过程 vs 结果：对推理与 agent 任务，只看最终答案不够，要评估推理过程、工具使用是否合理——与 PRM、agent 评估同源。
  • 动态/对抗评估：静态榜单容易被刷，转向动态生成题目、人类对战（如竞技场式 Elo）和对抗性红队测试。

为什么评估值得单列收尾

预训练的数据去污染、后训练的奖励设计、agent 的任务级评估、RAG 的检索质量度量——每一个方向的"前沿张力"最终都汇到"我们怎么知道它真的变好了"。评估是贯穿全栈、却最容易被低估的硬骨头。

08一表收束：六大板块如何咬合

预训练把能力压进参数 → 后训练 / RL 激发并对齐（RLVR 正把它从"对齐偏好"推向"对齐正确"）→ Agent 算法让能力在长链条任务里自主释放 → 多模态给它接上感知 → 搜推 / RAG 给它接上海量外部事实。全程被推理效率托底、被评估校验。
