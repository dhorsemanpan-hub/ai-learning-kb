01一个最大的困惑：算法和后端到底差在哪

这是所有疑问的根。把它讲透，后面全通。

很多人觉得"写代码的不都差不多吗"，其实两者面对的世界本质不同：后端面对的是一个"确定"的世界——同样的输入，必然得到同样的输出；而算法面对的是一个"不确定"的世界——他要驯服的是"模型"这个会临场发挥、还可能答错的大脑。

图 1 · 算法 vs 后端：一个造不确定的大脑，一个修确定的管道

一句话记住

后端的本质是"把确定的逻辑做得又稳又快"；算法的本质是"驯服模型这个不确定的大脑"。前者难在规模与稳定，后者难在让一个会犯错的东西变得可靠。

03Agent 工程师 ≠ 换皮后端，也 ≠ Agent 产品经理

你最纠结的点：既然照着 PRD 实现，那和后端有啥本质区别？

关键在于 Agent 工程师面对的不是确定的逻辑，而是一个会临场发挥、还可能犯错的大脑。他要解决的是"怎么让这个不靠谱的大脑稳定地把活干成、还不闯祸"——这是普通后端从不需要处理的难题。

图 3 · 后端、Agent 工程师、Agent 产品经理三栏对比

那 harness 到底是什么？

Agent 不是问一句答一句，而是一个"想 → 调工具 → 看结果 → 再想"的循环。harness（外层约束框架）就是裹在这个循环外面的那层控制——管循环何时停、管能调哪些工具、管出错怎么兜底、管什么操作必须踩刹车。它决定了这个大脑"敢不敢放它出去真的碰钱"。这正是 Agent 工程师的主战场，也是它和"照 PRD 填空"的根本差别：PRD 写不出"大脑突然乱来时怎么办"，这得靠工程师在不确定性里现场设计。

04数据为什么能让模型"变聪明"

你的疑问：数据怎么直接就让模型变聪明了？

因为模型本质只在做一件事——猜下一个字。但为了"猜得准"，它被迫学会了背后的一切知识。这就是数据能"喂出智能"的秘密。

图 5 · 数据如何把模型"逼"出智能

那 Evaluation（评估）算谁的活？数据 PM 又和算法怎么分？

评估其实有两层，千万别混为一谈：产品层评估看"用户/业务好不好用"，算法层评估看"模型准不准、用来指导下一轮训练"。同一份评估，PM 用它判断能不能上线，算法用它决定下一步怎么再训。

图 6 · 数据 PM、算法、工程在"数据 + 评估"上的分工

区分口诀

数据 PM 用嘴和文档说"该要什么数据、看什么指标"，不下场训练；算法工程师 才是真正把数据捏成模型能学的样子、并亲手把模型训出来的人。一个定方向，一个造东西。

06速查表：每个角色对"什么结果"负责

筛人时最该记住的一句话——看一个人，先看他对什么结果负责。

角色类别一句话职责对什么结果负责
业务总负责人业务定战略、拍资源营收、合规、整体成败
业务 PM产品定产品形态用户爱不爱用
Agent PM产品定能力与流程这 Agent 该会什么
数据 PM产品定要什么数据、看什么指标数据方向对不对
UI/UX 设计设计定长什么样好不好看好不好用
预训练算法算法造基座大脑模型底子聪不聪明
后训练算法算法调教成金融专才答得专不专业、听不听话
数据算法/工程算法把数据捏成模型能学的样子训练数据干不干净
搜推算法算法猜你想要、精准推推得准不准
Agent 工程师工程搭 harness、编排工具与记忆大脑放出去会不会稳、会不会闯祸
后端开发工程建账户/交易/高并发系统系统稳不稳、快不快
前端/客户端工程把界面做出来界面顺不顺
风控/合规保障划金融红线会不会违规闯祸
测试 QA / 运营保障把关 + 持续运营上线质量、长期留存

三句话彻底拎清

1. PM 类（业务/Agent/数据 PM）对"该做什么"负责——动嘴、动文档，不下场造。

2. 算法类（预训练/后训练/搜推/数据算法）对"模型这个不确定的大脑行不行"负责——这是这个 JD 的核心，别人无法替代。

3. 工程类（Agent 工程/后端/前端）对"能不能稳定地真跑起来"负责。其中 Agent 工程师之所以不是换皮后端，是因为他专门干一件传统后端没干过的脏活：驯服那个会临场发挥、还可能犯错的大脑。

筛人时一句话的判断口诀

  • 能讲清 "模型在 XX 上不行 → 分析是数据/方法问题 → 改了 XX → 指标涨了多少" 完整闭环 → 真算法
  • 只能讲 "我定义了产品要哪些能力、达到哪些指标" → 产品经理
  • 全在讲 "我把系统做得多稳多快、扛住多少并发" → 后端

这个 JD 要找的，是第一种人——能为"模型行不行"这件不确定的事负责，并用数据和训练方法把它一点点变行的人。
