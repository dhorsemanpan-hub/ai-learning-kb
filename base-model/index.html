00先搞清楚：什么是「基座模型」，为什么它最金贵

在拆流程之前，先把概念钉死，否则后面所有阶段都会含糊。

基座模型（Base Model / Foundation Model），指的是经过大规模预训练、但还没做任务特化微调的「裸模型」。它本质上只会做一件事：看着前面的文字，预测下一个词（token）。但就是这个看似简单到无聊的目标，在海量数据 + 巨大参数规模的加持下，把语法、常识、逻辑、代码、数学这些通用能力，全都「压」进了模型参数里。

一个关键事实：所有下游能力的天花板，都由基座模型决定。后训练（SFT / RLHF 那一套）做的是「从天花板往下取能力」，它无法创造基座里根本不存在的东西。基座推理弱，再怎么对齐也变不出强推理；基座金融知识少，后训练只能改表达方式，补不了底子。

用盖楼来类比整件事

基座模型 = 地基 + 建筑主体框架。后训练是精装修、摆家具、贴标识。装修再豪华，地基不牢就是危楼。而打地基是最贵、最耗时、技术门槛最高的环节——这也正是「有基座模型经验的人」最值钱的原因：会装修的人很多，会从零打地基并且打得稳的人，凤毛麟角。

01全流程总览：六个阶段 + 三道决策闸门

基座模型的制作不是「一条路走到底」，而是小步验证、随时可能回退的工程。先看一张全景图。

① 架构设计画图纸：定模型长什么样1–3 个月人力 + 小规模 ablation 算力
② 数据工程备料：把万亿 token 洗干净配好2–4 个月（可与①并行）存储 + 清洗算力，人力密集

⑤ 评测验收 + 全程体检贯穿全程人工评估成本
⑥ 优化与发布瘦身上线：量化蒸馏出报告2–4 周工程为主

为什么强调「随时回退」这件事

架构设计前会做小规模 ablation；预训练途中会跑 mid-training eval 提前发现问题；后训练完评测不过要回去补数据或调策略。每一次回退都在烧钱——一次完整预训练成本在百万到千万级。所以真正做过基模的人，最有价值的不是「会跑训练脚本」，而是他知道什么时候该叫停、什么时候该回头、回到哪一步。这个判断力是检验经验是否 solid 的核心。

02阶段一 · 架构设计：在烧第一张卡之前定好图纸

这是整个流程的「图纸阶段」。错了，后面几百万的算力全白费。

2.1 模型结构选型

当前主流基座几乎全是 Decoder-only Transformer，但内部有几处关键分化，每一处都是一次权衡：

Dense vs MoE
Dense＝每个 token 都过全部参数；MoE（混合专家）＝稀疏激活，每个 token 只过少数几个「专家」（Mixtral、DeepSeek-V3、Qwen-MoE 走这条路）。MoE 用「总参数很大、实际激活很少」冲规模，核心难题是 路由均衡（别让少数专家被挤爆）和 通信开销（专家分散在多卡上）。
注意力机制谱系
MHA（每个头独立 KV，最费显存）→ GQA（多个查询头共享一组 KV，省显存、几乎不掉点，当前最常用）→ MQA（所有头共享一份 KV，最省但易掉质量）→ MLA（DeepSeek 提出，低秩压缩 KV，极致省显存还保质量）。这个选择直接决定推理成本和长上下文可行性。
位置编码
主流是 RoPE（旋转位置编码），好处是支持长度外推。后面扩长上下文时，就是对 RoPE 做缩放/插值（NTK-aware、YaRN 等）。
激活函数
SwiGLU 已成标配——同等参数下比 ReLU/GELU 表现更好，代价是 FFN 层参数量略增。
归一化
RMSNorm + Pre-Norm 成为主流，比老的 LayerNorm + Post-Norm 训练更稳、深层梯度更健康。

2.2 超参数：每个数字背后都是千万级算力赌注

模型规模
参数量多大？1B / 7B / 14B / 72B？由 Scaling Law（规模规律） 决定「给定算力，参数多大 + 数据多少最优」。Chinchilla 之后大家倾向「小模型多喂数据」（过度训练），因为这样推理成本低、更划算。
深度 vs 宽度
层数和隐藏维度怎么配。深而窄泛化往往更好，但训练时跨层通信更复杂。
词表大小
影响压缩率和多语言覆盖，常见 32K–150K。中文/多语言要更大词表，但词表越大、embedding 层越占参数。
上下文长度
预训练通常先用较短（4K–8K）省算力，长上下文（32K/128K）留到后续持续预训练再扩。

2.3 Tokenizer 设计（容易被外行忽略，但极重要）

Tokenizer 决定文本怎么被切成模型能吃的最小单元，直接影响压缩率、多语言、以及数字/代码这类特殊内容的表现。

BPE / BBPE
字节对编码，最主流。从字符级开始，反复把高频组合「合并」成新词，生成词表。BBPE 在字节级操作，天然支持任意语言。
SentencePiece
Google 的 tokenizer 工具，支持 BPE 与 Unigram 两种算法，多语言友好，LLaMA 系列在用。
tiktoken
OpenAI 的高效 BPE 实现，Rust 写核心，速度快，GPT-4 系列在用。
数字 / 代码处理
数字是否逐位切分（影响算术能力）？代码缩进会不会被浪费成大量空格 token？这些细节直接影响下游能力——金融场景里金额数字的编码方式尤其要抠。

2.4 前置验证：用小钱排除不确定性

大规模训练前，必须用小模型 + 小数据先验证：

  • 架构 ablation：小规模上对比 Dense/MoE、不同头数、不同归一化，看谁更好
  • 数据配比 ablation：小模型试不同配比，看哪组下游任务最均衡
  • Scaling Law 外推：用 1B/3B/7B 的 loss 曲线拟合，预测 70B 要喂多少数据、何时到目标 loss
  • Tokenizer 评估：压缩率、多语言覆盖、数字编码质量

⚠️ 这一步的隐形价值＝省钱

一次完整预训练算力成本百万到千万级，而前置验证只花其百分之一。老手会在这一步尽量把不确定性榨干，而不是等大模型训完才发现「架构选错了」。

数据算法工程师（协同）— 提供配比初始建议与 ablation 数据集

训练基础设施工程师（协同）— 评估该架构在现有集群上的训练可行性、预设计并行策略

03阶段二 · 数据工程：决定上限的「备料」

行内话：数据决定上限，算法逼近上限。同样的架构，喂不同的数据，结果天差地别。

3.1 数据来源

来源内容典型规模主要注意点
Common Crawl互联网网页快照PB 级原始 / TB 级清洗后质量极度参差，需重度清洗
GitHub 等开源代码数百 GB许可证合规、去重
arXiv / 论文学术文献数十 GBLaTeX/公式需特殊处理
维基百科百科知识数十 GB高质量但量少
书籍版权书籍数十 GB版权是最大风险
金融/法律垂类行业语料视来源合规、PII（个人信息）是红线

3.2 数据清洗流水线（最脏最累，也最见功力）

一条语料从原始到能喂模型，通常要过 7–8 道关卡：

1

提取

从 HTML/PDF/文档里抠出纯文本。难点是表格、公式、多栏排版的正确还原（trafilatura / readability / PDF 解析器）。

2

语言识别与过滤

识别语种、过滤不需要的语言或低质混杂文本（fasttext 语言分类器）。

3

质量分类

用小模型（fasttext / 小 BERT）给文本打质量分，滤掉广告、垃圾评论、模板。有的团队用强模型标注高质语料来训这个分类器。

4

去重（最关键一步）

精确去重（MinHash + LSH）删完全重复段落；模糊去重（SimHash / 语义嵌入）删高度相似；文档级去重删整篇重复。去重不彻底，模型会「背答案」而不是「学规律」。

5

PII 脱敏

移除姓名、身份证、电话、邮箱等个人信息。在金融/医疗是法律红线，用正则 + NER 模型双保险。

6

有害内容过滤

滤掉色情、暴力、极端言论（分类器 + 规则组合）。

7

基准去污染（Decontamination）

把评测集（MMLU、HumanEval 等）的题目/答案从训练数据里精确剔除，否则模型「作弊」拿高分但真实能力不行。这步容易被忽略却极重要。

8

格式标准化

统一编码、换行、清理残留标记、规范公式表示。

3.3 数据配比：整个数据工程里最「玄学」也最核心的决策

配比的核心直觉

代码和数学被反复证明能提升通用推理能力——即便做中文模型也要喂大量代码。为什么？因为代码逼模型学「严格的因果与规则」，这种能力会迁移到自然语言推理上。

一个 70B 级模型的参考配比：网页 50–60% · 代码 15–25% · 数学 5–10% · 书籍 5–10% · 维基 3–5% · 垂类 2–5%。但没有万能配比，每个团队都要在自己的小规模 ablation 上验证。

3.4 合成数据：当真实语料见底

用强模型（如 GPT-4 级）生成数据来训自己的模型，是当前最热也最有争议的方向：

  • 优势：可控、能定向补短板、理论上可无限扩展
  • 风险：模型坍缩（Model Collapse）——用模型生成的数据训模型，分布会一代比一代窄，多样性塌掉
  • 最佳实践：合成数据补充而非替代真实数据；多源模型生成增加多样性；严格质量筛选

参与角色数据算法工程师（主）— 设计清洗管线、定配比策略、跑 ablation。产出：清洗管线代码、最终训练集、配比报告

数据工程师— 搭分布式数据加工基础设施（Spark/Ray 等）、保障存储与吞吐

领域专家（金融）— 审核垂类语料质量、给配比建议

04阶段三 · 预训练：烧钱的核心，最考验真功夫

把海量数据喂进模型，让它通过「预测下一个 token」学会一切。一次完整预训练成本百万到千万级，所以稳定性和效率压倒一切。

4.1 训练基础设施

GPU 选型
A100 80G / H100 / H800 是主流训练卡。卡间互联用 NVLink（机内）+ InfiniBand（机间），带宽是大规模并行的命脉。
集群规模
70B Dense 通常要数百到上千卡。MoE 因总参数大但激活少，往往要更多卡、但单卡计算量更低。

4.2 分布式训练策略（单卡装不下，必须拆）

实际是几种并行策略的组合，业内叫 3D / 4D 并行：

图 2 · 四种并行策略各管一件事，实战中叠加使用
Megatron-LM
NVIDIA 出品，最成熟的 TP+PP 实现，性能最优，但对模型代码侵入大。
DeepSpeed
微软出品，ZeRO 的开创者，侵入性小；纯 ZeRO 在超大规模下不如 Megatron 的 TP+PP 高效。
FSDP
PyTorch 原生 ZeRO 实现，易用灵活，极致优化略弱。
现实选择
多数团队用 Megatron + DeepSpeed ZeRO 混合方案取长补短，也越来越多基于 FSDP 做定制。

4.3 训练配置

学习率调度
标准做法：warmup（线性升温到峰值）→ cosine decay（余弦退火到最小值）。warmup 防初期梯度爆炸，decay 让后期「收细」。峰值 LR 常见 1e-4 ~ 3e-4，最小值约峰值 10%。
Batch size 渐增
初期用小 batch，逐步增大到目标值，帮助早期稳定。
权重衰减 & 梯度裁剪
前者防过拟合，后者（clip grad norm）防梯度爆炸，是稳定性基础保障。
数值精度
bf16 已成主流——比 fp16 动态范围大、不用 loss scaling、更稳。混合精度：前向 bf16，梯度/优化器状态 fp32。fp8 在兴起但还不成熟。
梯度检查点 / 重计算
前向不存全部中间激活，反向时重算。用计算换显存，大模型标配。

4.4 训练稳定性：Loss Spike 是头号敌人

🚨 Loss Spike（损失突然飙升）

大规模训练最常见也最致命的问题。诱因：LR 太高、混入异常样本、硬件故障导致梯度异常、MoE 路由突然偏斜。

预防：warmup + 梯度裁剪 + 数据把关 + 动态调 LR；检测：实时监控 loss，设阈值自动告警；恢复：回滚到最近 checkpoint、跳过问题数据、降 LR 继续。但回滚本身有代价（加载时间 + 数据偏移），所以老手会在前置验证阶段就尽量排除诱因。

4.5 训练过程监控（不是按下启动键就走人）

  • Loss 曲线：是否正常下降？有无 spike？训练/验证 loss 是否开始分离（过拟合信号）
  • Mid-training eval：途中定期跑下游 benchmark，看能力是否按预期「涌现」
  • Scaling Law 对比：实际 loss 是否贴合预测曲线？偏差大说明数据/架构有问题
  • 梯度范数 / 参数范数：是否稳定，突变常预示问题
  • 硬件监控：GPU 利用率、显存、通信占比、慢卡检测

4.6 课程学习与退火（不是一套配置跑到底）

课程学习
先喂「容易/通用/干净」的数据，后喂「难/专业/复杂」的，模仿人从易到难。怎么定义「难」本身是研究问题。
退火（Annealing）
训练最后 5–10% 的 token，用高质量数据替换混合数据、同时 LR 走到最低。这步对最终质量影响极大——相当于冲刺阶段用最精的粮。
长上下文扩展
预训练用 4K–8K，扩到 32K/128K 需持续预训练：对 RoPE 缩放（NTK-aware/YaRN）+ 长文档数据 + 序列并行。难点不是「塞进去」，而是塞进去后「中间信息会不会被忽略」（lost-in-the-middle）。

参与角色预训练算法工程师（主）— 定训练配置、监控全程、处理 spike、调策略。产出：base model checkpoint、训练日志、Scaling Law 拟合报告

训练基础设施工程师— 搭维护分布式集群、优化并行策略、保障 checkpoint 存储与网络。产出：训练框架配置、性能优化报告

数据算法工程师（协同）— 按 mid-eval 反馈调整数据配比

05阶段四 · 后训练 / 对齐：把「书呆子」调教成能干活

base model 是个「懂很多但不会好好干活」的状态——它会续写文本，但不会按指令答、不会拒绝不当请求、不会按格式输出。后训练就是教规矩。

5.1 SFT（监督微调）

SFT 的关键认知

用高质量「指令—回答」对，手把手教模型「这种问题就该这么答」。当前共识是质量远比数量重要：几千到几万条精品，胜过百万条噪声。因为 SFT 本质在教「格式和风格」，知识底子已由预训练决定。进阶手法 拒绝采样：让模型对一个指令生成多个答案，用评分模型挑最好的回灌训练——这是 SFT 通往 RL 的桥梁。

5.2 对齐方法谱系

RLHF（PPO）
经典路线：先训奖励模型（RM）→ 再用 PPO 优化策略。要同时维护 4 个模型，最重最不稳但上限最高。主要风险是 reward hacking——模型学会「骗」奖励而非真变好。
DPO
跳过显式 RM，直接用偏好对优化，把 RL 问题变成一个分类损失。轻、稳、好复现，工业界落地首选，但精细控制不如在线 RL。
GRPO
DeepSeek 提出，去掉 critic，用「一组采样的组内相对优劣」估计优势，省一半资源，训推理模型的热门。
RLVR
奖励来自可自动验证的信号（数学答案对不对、代码能否跑通），干净可无限生成，是推理模型爆发的核心。金融这类强正确/强合规场景特别契合。

5.3 安全对齐（金融场景的命根子）

  • Red Teaming：专门团队「攻击」模型诱导有害输出，发现的漏洞反哺训练
  • 安全 SFT 数据：教模型识别并拒绝不当请求
  • Constitutional AI：让模型用「宪法原则」自审自改输出
  • 金融特有红线：不承诺收益、不荐具体个股、不泄露用户金融信息——需专门构建安全训练数据

参与角色后训练算法工程师（主）— 构建 SFT 数据、训奖励模型、执行 RLHF/DPO/GRPO、迭代对齐。产出：aligned model、对齐报告

标注团队— 产出偏好标注、安全 red-team 数据

安全工程师（协同）— 定义安全边界、组织 red team、审核安全数据

06阶段五 · 评测：贯穿全程的决策依据

评测不是「最后贴个分数」，而是全程的体检表——决定能不能发布、哪块能力要补、该不该回退。

通用知识MMLU, MMLU-Pro污染严重，需去污染版本
推理GSM8K, MATH, ARC数学推理是能力硬指标
代码HumanEval, MBPP, LiveCodeBenchLiveCodeBench 动态更新，更抗污染
长上下文大海捞针, LongBench不只看「找不找得到」，还看中间/尾部信息利用率
指令遵循IFEval, AlpacaEval看是否严格按指令格式输出
安全TruthfulQA + 自建集金融合规需自建测试集

6.2 人工评测（补自动评测的盲区）

  • Arena 式盲评：两模型匿名对决，人类选更优者（Chatbot Arena 那套）
  • 领域专家评估：金融场景要从业者审专业性与合规性
  • LLM-as-Judge：用强模型当评审，便宜但自带偏置（偏好更长、更像自己的输出）

🚨 基准污染（Benchmark Contamination）

训练数据混入了评测题/答案，模型「背了答案」拿高分但真实能力不行——这是当前评测体系最大的信任危机。对策：训练数据去污染、用动态更新的 benchmark、自建不公开的内部评测集、对比「污染题 vs 干净题」的得分差。

参与角色评测工程师（主）— 搭自动化评测管线、维护 benchmark 套件、跑分出报告。产出：评测报告、能力雷达图、回归结果

领域专家（金融）— 审金融专业题的质量与答案标准

07阶段六 · 优化与发布：瘦身上线

训好不等于能上线——还得让它跑得起、跑得快、跑得省。

7.1 模型效率优化

量化 Quantization
GPTQ（训练后量化）、AWQ（激活感知）、GGUF（llama.cpp 格式，支持 CPU）。INT4 量化通常只损失 1–2% 能力，但速度提升显著。
蒸馏 Distillation
用大模型（教师）指导小模型（学生），让小模型逼近大模型能力。DeepSeek-R1 把 671B 的推理能力蒸馏到 7B/14B 就是典型。
KV Cache 优化
PagedAttention（vLLM）分页管理 KV、消除显存碎片；架构层用 GQA/MLA 减少 KV；还可量化 KV。
投机解码
小模型快速「猜」后续 token，大模型并行验证；猜对就加速，猜错回退。在代码/数学这类确定性高的场景加速最明显。

7.2 模型卡与技术报告

  • Model Card：能力边界、训练数据概况、已知局限、使用许可
  • Technical Report：架构、训练方法、评测、安全评估（Llama/Qwen/DeepSeek 的报告是范本）
  • 开源协议：Apache 2.0 / MIT / 自定义（商用限制、输出归属等）

参与角色推理优化工程师（主）— 量化、蒸馏、推理框架优化。产出：优化后权重、推理性能报告

算法工程师（协同）— 评估优化后能力损失是否可接受

基模团队往往 40+ 人，但核心可归为「定方向 / 造能力 / 保运行」三类。下表只列与基模制作直接相关的角色。

招聘视角的关键提醒

「有基座模型经验」是个很宽的说法。同样号称做过基模，架构、数据、预训练、后训练四个工种的能力栈差别极大。你要先搞清业务真正缺哪一段——是缺能定架构、跑预训练的「打地基的人」，还是缺后训练对齐的人。下一节给你按工种拆开的检验方法。

09人选检验建议：怎么判断基模经验是否 solid

核心是分辨「真正主导过决策」还是「只在周边打杂、跑过脚本」。

一句话总判断口诀

真正做过基模的人，能讲清这条完整闭环：「我面对什么问题 → 我做了什么决策 → 为什么这么选 → 结果怎样 → 如果重来我会怎么改」。只会说「我参与了训练」却讲不清决策逻辑的，大概率是执行者而非负责人。

9.1 架构设计

🟢 「我对比了 Dense 和 MoE 在 XX 规模下的 loss 与推理成本，最终选 XX，因为……」
🟢 「Tokenizer 词表大小我们做过消融，XX 大小在中文压缩率和参数效率上最优」
🔴 「架构是 leader 定的，我负责实现」——没有决策参与
🔴 「我们用的就是 LLaMA 架构」——只说结果，讲不出为什么

9.2 数据工程

🟢 「配比从 40% 代码起调，发现超过 25% 后通用能力反降，最后定在 18%」
🟢 「去重用 MinHash+LSH，阈值 XX，去重率 XX%，对 loss 的影响是……」
🟢 「合成数据我们做了分布对比，发现 XX 维度会坍缩，所以只用了 5%」
🔴 「数据是数据团队准备的，我只管训练」——核心能力缺失
🔴 「清洗就是去重和过滤」——说不清具体做法、阈值怎么定

9.3 预训练

🟢 「训到 40% 时出现 loss spike，我判断是 XX 数据导致，回退到 XX checkpoint 并跳过那批数据」
🟢 「Scaling Law 预测 70B 要训 XX token，实际在 XX 时 loss 偏离预测，我分析原因是……」
🟢 「并行用 TP=8 + PP=4 + ZeRO-3，通信占比 XX%，瓶颈在 XX，我们这样优化……」
🔴 「我负责盯训练曲线」——只监控不决策
🔴 「训练用 Megatron 跑的」——只说工具，讲不出策略
🔴 「训练一直很稳，没出过问题」——要么规模太小，要么没真正负责过

9.4 后训练 / 对齐

🟢 「我们从 PPO 切到 DPO，因为 PPO 在 XX 上 reward hacking 严重，切完 XX 指标提升 XX」
🟢 「奖励模型我们用过程奖励 PRM 而非只看结果，在数学题上对齐效果更好」
🔴 「我做了 SFT，把数据喂进去微调」——讲不清数据怎么构造、质量怎么控
🔴 「对齐就是 RLHF」——背概念，落不到自己做过的决策

9.5 跨阶段的「资深度」信号

  • 谈成本意识：能自然提到算力预算、何时该停、回退代价——说明真烧过钱
  • 谈失败经验：能讲清楚「踩过的坑、走过的弯路」，比只讲成功更可信
  • 谈取舍而非堆名词：资深的人讲的是 trade-off（为何选 A 不选 B），新手堆术语
  • 能外推到没做过的规模：「我做过 7B，但我知道到 70B 时 XX 会变成瓶颈」——体现对原理的理解，而非死记流程
  • 对数据的敬畏：顶尖的预训练人都会强调「数据是上限」，张口只谈模型结构、不谈数据的，往往深度不够

给你筛人时的最后提醒

顶会论文数量（JD 里说的 5–10 篇）主要对应架构 / 预训练方向——因为这块创新最容易出论文。但要注意：论文多 ≠ 工程落地强。能发论文的人未必扛过千卡集群的 loss spike，扛过大规模训练的人未必爱发论文。判断时把「学术产出」和「工程实战」分开看，再对照业务到底要哪一种。
