大模型"算法"到底分几种
每一种在做什么、怎么招

01先建坐标系：算法不是铁板一块

"算法工程师"是个大筐，里面装着差别极大的几种人。先有张全局地图。

上一册我们说"算法 = 驯服模型这个不确定的大脑"。但造大脑、教大脑、放大脑出去干活是完全不同的手艺，再叠加"它处理文字还是图片、是搜索推荐还是自主决策"，就分出了好几个工种。理解算法的第一步，是看懂这两条轴：纵向 = 模型的生命周期阶段，横向 = 它处理什么数据/解决什么问题。

图 1 · 理解算法的两条轴：生命周期阶段 × 技术领域

一句话记住

别问"他是不是算法"，要问"他在生命周期的哪一段、处理哪种数据"。一个训基模的人和一个做搜推的人，虽然都叫算法，技术栈和思维方式可能天差地别。

03先排个雷：传统机器学习 ≠ 大模型算法

这是筛简历最容易踩的坑——同样写"算法/机器学习"，可能是两套完全不同的体系。

很多简历写着"机器学习算法工程师""8 年算法经验"，但他做的可能是传统机器学习——靠人工设计特征、训练小模型做风控评分、点击率预测。这套手艺很值钱，但和你要的大模型（LLM）算法是两个时代的东西。看不清，就会把"搜推老兵"误当成"基模新锐"。

图 3 · 两套算法体系对比：特征工程小模型 vs 预训练大模型

筛简历判断口诀

看到 XGBoost / LightGBM / 特征工程 / CTR / 评分卡 为主 → 偏传统 ML（搜推、风控背景）。看到 预训练 / Transformer / SFT / RLHF / 千亿参数 为主 → 才是大模型算法。两者不是高低，是不同赛道；本 JD 主要要后者。但注意：搜推岗往往两套都懂，是天然的过渡人才。

04基模算法（预训练）：造大脑的人

最上游、最烧钱、最容易出顶会论文，也是本 JD 最想补强的方向。

做什么：从零把一个通用大模型训出来——决定模型多大、用什么数据、怎么配比、训练怎么更高效不崩。这是"打地基"的活，地基决定了整栋楼的上限。
适用业务：不直接对某个业务，而是产出"底座大脑"，供下游所有金融场景共用。JD 里"我们需要更精细化地把基模做起来"指的就是这群人。

技术栈名词（大白话版）

Transformer
几乎所有大模型的"通用发动机"骨架。你不用懂内部，但候选人必须张口就能聊它的变体和取舍。
Token / Tokenizer
模型不认字，先把文字切成一小块一小块的"积木"（token）。Tokenizer 就是那把切积木的刀，切法好坏直接影响中文和数字处理。
参数量
模型"脑容量"大小，常说的 7B、72B 就是 70 亿、720 亿个旋钮。越大越聪明，但越贵越难训。
Scaling Law
"规模定律"——多大模型、多少数据、多少算力能换来多少智能，有可预测的规律。预训练高手靠它决定怎么花钱最划算。
MoE 混合专家
把一个大模型拆成很多"专科医生"，每次只叫醒相关的几个，省算力又保能力。当下基模的主流架构之一。
分布式 / 并行训练
一张卡装不下、算不动，得几百上千张显卡协同干一件事。怎么切分、怎么不互相等待，是硬功夫。
数据配比
中文多少、代码多少、数学多少、金融语料多少——这个"配方"极大影响模型擅长什么。行内说"数据决定上限"主要指这里。

日常工作内容

  • 设计/改进模型结构和训练方法，跑大规模实验，盯着损失曲线调参；
  • 清洗、配比、去重海量训练数据（脏活但极关键）；
  • 解决训练中途崩溃、变慢、算力浪费等工程难题；
  • 读最新顶会论文、复现、把好点子用进自家模型。

招聘判断口诀

真预训练高手能讲清 "我们当时数据配比怎么定、为什么这么定、改了之后哪类能力涨了"，能聊 Scaling Law 和架构取舍。顶会论文 5-10 篇在这个方向是合理硬通货——因为预训练创新最容易发论文。只会"调用 API、做提示词"的，不是这一档。

05后训练算法 + 强化学习 RL 专题：教大脑的人

基模是"博学但不会做人"的通才，后训练把它调教成"听话、专业、不乱说"的金融专才。RL 是这一步越来越核心的武器。

做什么：在已有基模上做精调和对齐，让它听指令、符合人类偏好、在金融场景里既专业又守规矩。
适用业务：几乎所有要落地的金融 AI 都离不开——让通用大脑变成"懂理财、会答客户、不乱承诺收益"的专家。JD 说"之前主要做金融后训练落地"就是这层。

图 4 · 后训练三件套：SFT → RLHF/DPO → 奖励模型

技术栈名词（大白话版）

SFT 监督微调
给模型看大量"好问题—好答案"的范例，让它照着学会某种风格或专业能力。最基础的教法。
RLHF
"用人类反馈做强化学习"。不直接给答案，而是对模型的多个回答排好坏，让它自己往人喜欢的方向进化。
DPO
RLHF 的简化版，更省事、更稳。同样是"喜欢这个不喜欢那个"，但训练流程更短，近两年很流行。
奖励模型 RM
一个专门"当裁判"的小模型，替人类给海量回答自动打分。没有它，RL 没法大规模跑。
PPO / GRPO
两种具体的强化学习算法（"怎么根据打分去更新模型"的配方）。GRPO 是近期更高效的新宠，候选人聊得出区别说明在一线。
对齐 Alignment
让模型的行为"对齐"人类期望——有用、诚实、无害。金融场景里还要叠一层"合规对齐"。
幻觉
模型一本正经地胡说八道、编造不存在的数据。金融里这是大忌，后训练的重要目标就是压幻觉。

招聘判断口诀

真后训练高手能讲清 "模型有个毛病 → 我用 SFT/DPO/RL 哪种手段、配了什么数据 → 这个指标降了多少" 的完整闭环。会聊 RL（PPO/GRPO）、自己搭过奖励模型的，是当下最抢手的一档——因为这正是从"能用"迈向"好用"的关键，也是各家拉开差距的地方。

06Agent 算法：教大脑"自己干活"的人

注意：这里说的是 Agent 算法，和上一册讲的 Agent 工程是两拨人、两件事。

做什么：让模型本身学会"自主完成多步任务"的能力——会拆解目标、会决定该调哪个工具、做错了会反思纠正。这是在"训练模型的脑回路"，让它天生擅长当 Agent。
和 Agent 工程的区别：算法侧训"大脑会不会规划用工具"，工程侧（上一册的 harness）搭"外面的笼子和管道让它安全落地"。一个改大脑，一个搭环境。

图 5 · Agent 算法 vs Agent 工程：改大脑 vs 搭环境

技术栈名词（大白话版）

Function Calling
让模型学会"该用工具时，吐出一份格式规范的调用指令"。是 Agent 能动手做事的基本功。
ReAct
"边想边做"的范式：想一步→做一步→看结果→再想。让模型像人一样推进多步任务。
CoT 思维链
让模型"把推理过程一步步写出来"再给答案，复杂问题更不容易错。算 Agent 的基础能力。
Agent 记忆
让 Agent 记住之前发生过什么、用户是谁、做到哪一步。算法侧研究怎么记得准、不串台。
RL for Agent
用强化学习直接训练"做整个任务"的能力——任务成功就奖励。当下让 Agent 变强的最前沿手段。

招聘判断口诀

Agent 算法候选人应能讲 "我怎么训模型让它在多步任务上成功率更高"（RL、轨迹数据、反思机制）。如果他只讲"我写提示词、连了几个工具、调通了流程"——那更偏 Agent 工程 / 应用，不是算法。本 JD 的"agent 工程"那条更看重绩效好、做过非内部系统的 harness，属工程侧；若能兼具 Agent 算法能力则是稀缺加分。

07多模态算法：让大脑长出眼睛和耳朵

前面说的基模多是纯文字，多模态算法让模型能同时看图、听音、读视频。

做什么：训练模型理解或生成多种形态的内容——看懂一张财报截图、识别身份证、读懂 K 线图、甚至生成讲解视频。核心难点是让"图像/语音"和"文字"在模型里对齐到同一个理解空间。
金融场景适用性：票据/合同/身份证识别、研报图表解读、客服语音、视频化投教内容——多模态在金融里落地点很多，但相比文本基模，目前是"重要但更偏特定场景"的方向。

技术栈名词（大白话版）

ViT
"视觉版 Transformer"，把图片也切成小块当 token 处理，让大模型能像读文字一样读图。
CLIP
让"图"和"描述它的文字"在模型里靠得很近的一种训练法。是图文互通的基石。
扩散模型 Diffusion
主流的"文生图/视频"技术：从一团噪点一步步"去噪"还原出清晰图像。AI 画图、生成视频靠它。
多模态对齐
把图像编码器和语言大脑"焊"在一起，让模型看完图能用语言准确说出来。多模态训练的核心活。

招聘判断口诀

多模态候选人要能讲清 "图像怎么编码、怎么和文本对齐、训练数据怎么配"。注意区分两类：做理解（看图答题）和做生成（文生图/视频）技术栈有别。结合本 JD，金融更常用"理解类"多模态（证件、票据、图表）；纯生成方向匹配度相对低。

08搜推算法：最老牌、也最懂"把对的东西端到你面前"

JD 把搜推列为相关方向。它既是传统 ML 的主场，也正在和大模型快速融合。

做什么：你搜一下给最相关结果（搜索），你没搜也能猜你想要并推给你（推荐）。在金融里就是"推匹配你风险偏好的理财/贷款产品""搜一下基金给最相关的"。
和大模型的融合：传统搜推靠"召回+排序"两段式；大模型时代越来越多用向量检索和 RAG，让推荐更懂语义、更像对话。

技术栈名词（大白话版）

召回 / 排序
搜推的两段式：先从百万商品里粗筛出几百个"可能相关"（召回），再精挑细排出最该给你看的几个（排序）。
Embedding 向量
把一段文字/一个商品变成一串数字坐标，意思相近的坐标就靠得近。是"语义搜索"的基础。
向量检索
在海量向量里快速找出"意思最接近"的那些。让搜索从"对字面"升级到"对意思"。
双塔模型
把"用户"和"商品"各自编码成向量，看谁和谁更配。推荐系统的经典结构。
RAG 检索增强
回答前先去知识库检索相关资料，再让大模型基于资料作答。让大模型不靠瞎编、能引用最新/专有数据。金融问答里极常用。

招聘判断口诀

搜推候选人能聊 召回排序、双塔、向量检索、A/B 实验、点击/转化指标。同时懂传统搜推又会 RAG/向量化的人，是连接传统与大模型的稀缺过渡人才，对金融搜推场景很有价值。

09能力分级 · 人才地图 · 速查表

最后落到最实操的：怎么判断深浅、从哪挖人、一张表速查。

9.1 能力分级：硬通货怎么读

层级典型表现怎么判断
初级能跑通别人的训练/微调流程，按指导调参、清数据讲得清流程，但讲不清"为什么这么设计"
中级能独立负责一个方向，定方案、定数据策略、出指标能讲清完整闭环：问题→方案→效果，有自己的判断
资深能定整个模型/方向的技术路线，带人，有标志性产出有"代表作"：主导过知名模型、核心论文、或重大业绩

硬通货的正确读法

顶会论文（NeurIPS/ICML/ICLR/ACL/CVPR 等）：预训练、多模态方向 5-10 篇是合理硬通货，看一作和引用；榜单/比赛（Kaggle、各类 benchmark 榜首）：说明实战调优能力强；落地业绩：Agent/后训练/搜推更该看"上线了什么、指标涨了多少、扛了多大流量"。不同方向看不同硬通货——别拿论文数去要求 Agent 工程的人。

9.2 人才地图：各类算法从哪挖、有多稀缺

方向稀缺度从哪些团队挖（参考）
基模 / 预训练极稀缺头部基模团队：阿里千问、字节、百度文心、月之暗面、DeepSeek、智谱等
后训练 / RL很稀缺同上各基模团队的对齐/后训练组；做过 RLHF 落地的尤其抢手
Agent 算法较稀缺头部 Agent 产品团队、做过 coding/通用 agent 的团队
多模态较稀缺多模态基模团队、视觉/语音强的研究院
搜推相对充足各大厂搜索推荐部门，人才池大，会 RAG 的更值钱

结合本 JD 的策略

JD 点名"通用基模算法的人（阿里夸克、千问 toC、字节、百度文心）"——精准对应极稀缺的基模/预训练池。这类人薪资高、流动谨慎，建议"优质优先、强将与潜力均可"，并用"应用在元宝/微信/WorkBuddy 等真实金融场景"作为吸引点。后训练和 Agent 可适当放宽到"有强落地业绩"的候选人。

9.3 速查表：听到这句话 → 大概率是哪类算法

候选人这么说大概率是本 JD
"我研究数据配比、架构、Scaling Law，发了 N 篇顶会"基模 / 预训练最看重

"我做 XGBoost、特征工程、评分卡、CTR"传统 ML需甄别

全册最后一句话

判断一个算法候选人，永远先问两件事：① 他在模型生命周期的哪一段（造/教/放出去）？② 他处理哪种数据、解决哪类问题（文本/多模态/搜推/RL）？ 落进格子，再看硬通货和闭环讲述能力。本 JD 要的核心，是能为"基模和后训练这个不确定的大脑到底行不行"负责的人。
